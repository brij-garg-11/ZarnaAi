The Core Idea

Right now the bot sends replies but you get no signal on how those replies performed. Adding an analytics layer turns every message exchange into training data — not for the AI model itself, but for you to know exactly which bot behaviors drive engagement vs. kill it.

The Meta Analogy

Instagram tracks: did you keep scrolling after this post? Did you DM someone about it? Did you visit a profile? Every interaction is a signal that feeds back into what gets shown next. We do the same thing for SMS conversations. Did the fan reply after this bot text? How fast? Did they click the link? Did they go quiet? Every bot message has an engagement outcome — we just need to start recording it.

Database Schema Changes

Three additions to your existing Postgres database. All additive — existing tables untouched except for new columns on messages.

1. Augment the existing messages table

ALTER TABLE messages (new columns)

-- Add to _MIGRATIONS in postgres.py (idempotent, safe to redeploy)
ALTER TABLE messages
  ADD COLUMN IF NOT EXISTS intent              TEXT,
  ADD COLUMN IF NOT EXISTS tone_mode           TEXT,
  ADD COLUMN IF NOT EXISTS routing_tier        TEXT,
  ADD COLUMN IF NOT EXISTS reply_length_chars  INT,
  ADD COLUMN IF NOT EXISTS has_link            BOOLEAN  DEFAULT FALSE,
  ADD COLUMN IF NOT EXISTS conversation_turn   INT,
  ADD COLUMN IF NOT EXISTS gen_ms              FLOAT,
  ADD COLUMN IF NOT EXISTS -- engagement outcomes (backfilled async)
                            did_user_reply      BOOLEAN,
  ADD COLUMN IF NOT EXISTS reply_delay_seconds INT,
  ADD COLUMN IF NOT EXISTS went_silent_after   BOOLEAN,
  ADD COLUMN IF NOT EXISTS link_clicked_1h     BOOLEAN,
  ADD COLUMN IF NOT EXISTS msgs_after_this     INT;

The context columns (intent, tone, routing_tier) are written at reply time. The outcome columns (did_user_reply, reply_delay_seconds, etc.) are backfilled by a background job after the next fan message arrives.

2. New conversation_sessions table

CREATE TABLE conversation_sessions

CREATE TABLE IF NOT EXISTS conversation_sessions (
  id                    BIGSERIAL PRIMARY KEY,
  phone_number          TEXT    NOT NULL REFERENCES contacts(phone_number),
  started_at            TIMESTAMPTZ DEFAULT NOW(),
  last_active_at        TIMESTAMPTZ DEFAULT NOW(),
  ended_at              TIMESTAMPTZ,
  user_message_count    INT     DEFAULT 0,
  bot_message_count     INT     DEFAULT 0,
  ended_by              TEXT,    -- 'user_silence' | 'user_stop' | ongoing=NULL
  came_back_within_7d   BOOLEAN,
  first_intent          TEXT     -- what was the fan's opening intent?
);

CREATE INDEX IF NOT EXISTS idx_sessions_phone
  ON conversation_sessions(phone_number, started_at DESC);

CREATE INDEX IF NOT EXISTS idx_sessions_ended
  ON conversation_sessions(ended_at, ended_by)
  WHERE ended_at IS NOT NULL;

A "session" = contiguous conversation window. A new session starts when a fan messages after 24h of silence. Link the message rows to a session via the session ID.

3. New bot_reply_analytics view (computed, no extra storage)

CREATE VIEW bot_reply_analytics

CREATE OR REPLACE VIEW bot_reply_analytics AS
SELECT
  intent,
  tone_mode,
  routing_tier,
  COUNT(*) AS total_bot_messages,
  ROUND(AVG(did_user_reply::int) * 100, 1)   AS reply_rate_pct,
  ROUND(AVG(went_silent_after::int) * 100, 1) AS dropoff_rate_pct,
  ROUND(AVG(link_clicked_1h::int) * 100, 1)  AS link_ctr_pct,
  ROUND(AVG(reply_delay_seconds), 0)          AS avg_reply_delay_s,
  ROUND(AVG(msgs_after_this), 1)              AS avg_msgs_after,
  ROUND(AVG(reply_length_chars), 0)           AS avg_bot_reply_length
FROM messages
WHERE role = 'assistant'
  AND did_user_reply IS NOT NULL  -- only scored rows
GROUP BY intent, tone_mode, routing_tier
ORDER BY reply_rate_pct DESC;

This view is the core of your analytics dashboard. Query it and you immediately know which intent+tone combinations drive the most continued conversation vs. drop-off.

Change 2: New outcome_scorer.py — backfill signals when next message arrives

app/analytics/outcome_scorer.py (new file)

Called at the top of every inbound message handler — scores the previous bot reply now that we know the fan replied (or didn't):

def score_previous_bot_reply(phone_number: str, new_user_msg_time) -> None:
    """
    When a fan sends a new message, we now know the outcome of the
    previous bot reply. Update that row with engagement signals.
    Fire-and-forget — never blocks reply generation.
    """
    conn = get_db()
    with conn.cursor() as cur:
        # Find the last bot message for this fan
        cur.execute("""
            SELECT id, created_at FROM messages
            WHERE phone_number = %s AND role = 'assistant'
            ORDER BY created_at DESC LIMIT 1
        """, (phone_number,))
        last_bot = cur.fetchone()
        if not last_bot: return

        bot_id, bot_time = last_bot
        delay = int((new_user_msg_time - bot_time).total_seconds())

        cur.execute("""
            UPDATE messages SET
                did_user_reply      = TRUE,
                reply_delay_seconds = %s
            WHERE id = %s
        """, (delay, bot_id))
    conn.commit()

A separate daily cron job handles the "went_silent_after" backfill for messages that never got a reply — looks for bot messages older than 24h with no subsequent fan message.

Drop-off trigger patterns — what kills conversations

These are example patterns that emerge when you look at the last bot message before silence. Real data will be specific to Zarna's responses, but these types of patterns appear in every bot:

  • Long replies (300+ chars): Drop-off rate spikes when bot sends a wall of text — fans feel lectured, not chatted with
  • Too many questions in one message: "What city are you in? Have you been to a show before? What's your favorite bit?" — overwhelming, fans freeze
  • Deflection phrasing: "I don't have info on that" or generic fallback responses kill engagement immediately
  • Back-to-back links: Sending a link two messages in a row — fans feel sold to and leave
  • Tone mismatch: Fan shares something vulnerable, bot replies with a joke — immediate conversation ender

The Feedback Loop — How It Gets Smarter

Data collection is only valuable if it feeds back into improving the bot. Here's the full intelligence loop.

Short-term loop (manual, weekly)

Review the analytics dashboard every week. Look at:

  • Which intents have high drop-off this week?
  • Any specific bot replies with 0% continued conversation?
  • Are link CTRs going up or down after voice config changes?

→ Tweak creator_config/zarna.json based on what you find. This is the weekly Brij workflow.

Medium-term loop (automated alerts)

Build simple threshold alerts:

  • Drop-off rate this week > 35% → Slack alert to Brij
  • A specific intent's reply_rate drops > 15% from last week
  • Link CTR drops below 5% in last 48h

→ Add a Railway cron job that runs these checks nightly and posts to a Slack webhook.

Long-term loop (A/B testing bot variants)

Once you have enough signal, run soft experiments:

  • For 10% of SHOW intent messages, use shorter replies — does CTR improve?
  • Test ending general replies with a question vs. not — does it increase session depth?
  • Test different link placement (at end vs. mid-message) — which gets more clicks?

→ Track by variant in the messages table. You're doing real algorithm optimization now.

Future loop (LLM-assisted optimization)

Once you have 10k+ scored messages, you can feed the analytics to an LLM and ask it:

  • "Here are the top 20 bot replies by reply_rate. What do they have in common?"
  • "Here are the 20 worst drop-off triggers. What patterns do you see?"
  • "Suggest voice config changes based on this week's engagement data"

→ Literally paste the analytics output into Claude or GPT. This is the automated Brij workflow of the future.

What you can immediately optimize with this data
  • Reply length: Find the optimal character count for maximum continued engagement. (Hypothesis: 80–140 chars wins every time)
  • Question ending: Does ending a bot reply with a question increase session depth? Data will answer this definitively.
  • Link timing: Which conversation turn has the highest CTR? Don't push links on turn 1 if turn 3 converts 3x better.
  • Tone matching: When fans signal vulnerability, does the bot actually respond with the vulnerable tone? Is the tone classifier doing its job? Check with data.
  • Session starters: What does the bot say in reply #1 that leads to 10+ message sessions vs. 1-and-done? That's your most valuable signal.

Build Phases

Ordered by value delivered vs. effort. Start with Phase 1 — you'll have real signal data within days of deploying.

Phase 1 — Data foundation (1–2 days) highest priority
1

Add columns to messages table

Migrate intent, tone_mode, routing_tier, reply_length_chars, has_link, conversation_turn, gen_ms onto the messages table. These are all values already computed in handler.py — just need to persist them. 1 migration + 1 save_reply_context() method in postgres.py.

2

Write context at reply time in handler.py

After save_message(assistant, reply), call save_reply_context() with the intent/tone/tier/gen_ms values. Fire-and-forget in the executor so zero latency impact on reply.

3

Score previous bot reply on each inbound message

At the start of handle_incoming_message(), before anything else, fire a background thread that updates the previous bot reply row with did_user_reply=TRUE and reply_delay_seconds. This is retroactive scoring — every new fan message scores the bot's last reply.

4

Daily "silence" backfill cron job

A Railway cron job runs every night. For each bot reply older than 24h with did_user_reply IS NULL, set did_user_reply=FALSE and went_silent_after=TRUE. This fills in the drop-off signal.

Phase 2 — Analytics API (2–3 days) high value
5

Create bot_reply_analytics view in Postgres

The SQL view from the Schema tab. Runs on the DB side — no application code needed. Query it and you get reply_rate, dropoff_rate, link_ctr broken down by intent + tone in one fast query.

6

Add /analytics/* API endpoints

New Flask blueprint in app/analytics/blueprint.py, protected by the existing admin auth. Expose the engagement summary, intent breakdown, drop-off triggers, and session depth histogram.

7

Add an Analytics tab to the admin dashboard

Extend the existing admin UI with a new tab showing the core metrics. Charts for reply_rate by intent, session depth histogram, link CTR trend. Could be a simple HTML/JS tab using the existing admin_bp structure.

Phase 3 — Conversation sessions (3–5 days) medium priority
8

Create conversation_sessions table and session manager

Add the table from the Schema tab. Write a SessionManager class that finds or creates a session for an inbound message (24h gap = new session). Track session depth and ended_by signals.

9

Session-level metrics in dashboard

Avg session depth, session duration distribution, 7-day return rate. These answer "are fans coming back?" which is the core retention metric for a subscription text service.

Phase 4 — Feedback loop automation (ongoing) future
10

Automated weekly digest

A Railway cron job emails Brij every Monday with: top 5 performing bot reply patterns, top 5 drop-off triggers, link CTR for the week, and session depth trend vs. prior week.

11

A/B variant tracking

Add a variant column to the messages table. When testing a voice config change, tag those sessions with a variant ID. Compare engagement metrics between control and variant groups.

12

LLM-assisted pattern analysis

Build a script that pulls the top/bottom performing bot replies from the DB and feeds them to the Gemini API with a prompt like: "What patterns make these high-performing? What changes should we make to the voice config?" Turns data into actionable recommendations.
