# Bot Creation Pipeline
_How a new creator goes from sign-up to a live, personalized AI texting bot_
_Last updated: Apr 22, 2026_

---

## Overview

The goal is a fully self-serve pipeline where any performer or business fills out a form, and within ~70 seconds they have a dedicated phone number with an AI bot that sounds like them, knows their content, and can answer fan questions using their actual data.

Zarna is the proof of concept. Every new creator (HaleyBot, etc.) follows this same pipeline automatically. Zarna becomes just another slug — no special casing anywhere.

---

## Build Philosophy

**Do phases. Test each one. Move on only when it passes.**

Each phase is independently testable before the next one starts. Nothing gets pushed to production until you explicitly say so. The local chat simulator (Phase 6) runs throughout development so you can validate bot behavior at any point without needing Twilio or a real phone number.

---

## What Gets Created Per Creator

```
operator_users row
  └── creator_slug, account_type, phone_number (filled after provisioning)

bot_configs row
  └── config_json (from onboarding form), status, error_message

creator_configs row (Postgres — NOT a file on disk)
  └── config_json JSONB — AI personality: voice, style rules, guardrails, tone examples
      Generated by LLM from their form data
      Survives every Railway redeploy because it lives in the DB, not the filesystem

creator_embeddings rows (Postgres, pgvector)
  └── All their content as vector chunks
      Scraped from website, uploaded docs, podcast feeds, bio text
      Scoped by creator_slug — impossible to retrieve another creator's data

contacts table
  └── Empty at start — fans added as they text in
```

> **Why Postgres and not files?** Railway containers are ephemeral — every deploy wipes
> the local filesystem. Config files written at runtime (like `creator_config/haley.json`)
> would be destroyed on the next code push. Both configs and embeddings must live in the
> database to survive deploys.

---

## Status Lifecycle

```
submitted → provisioning → live
                ↓
         provisioning_failed  (error_message stored, retry available)
```

| Status | What it means | Frontend shows |
|---|---|---|
| `submitted` | Form saved, provisioning not started | "We're setting things up…" |
| `provisioning` | Background job running | Spinner: "Setting up your bot…" |
| `live` | Everything done, bot is active | Dashboard with phone number |
| `provisioning_failed` | A step broke | "Setup hit a snag — retry" button |

## Phase 1 — Database Foundation
**Time: ~30 min | Risk: None | Touches: Railway console + `operator/app/db.py`**

### Step 1A — Enable pgvector on Railway Postgres

Open Railway → your project → Postgres service → Data tab → Query runner. Run:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

This is a one-time command. It unlocks the `vector` column type and the `<=>` cosine similarity operator.

### Step 1B — Add new tables to `operator/app/db.py`

Add to the `statements` list in `init_db()`:

```sql
-- Personality configs — one row per creator, survives redeploys
CREATE TABLE IF NOT EXISTS creator_configs (
    id           BIGSERIAL PRIMARY KEY,
    creator_slug TEXT UNIQUE NOT NULL,
    config_json  JSONB NOT NULL DEFAULT '{}',
    created_at   TIMESTAMPTZ DEFAULT NOW(),
    updated_at   TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_cc_slug ON creator_configs(creator_slug);

-- Multi-tenant RAG embeddings
-- Dimension = 3072 to match gemini-embedding-001 (the model Zarna's
-- existing training data uses). ivfflat caps at ~2000 dims so we cast
-- to halfvec(3072) for the HNSW index.
CREATE TABLE IF NOT EXISTS creator_embeddings (
    id           BIGSERIAL PRIMARY KEY,
    creator_slug TEXT NOT NULL,
    chunk_text   TEXT NOT NULL,
    source       TEXT NOT NULL DEFAULT 'general',
    embedding    vector(3072),
    created_at   TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_ce_slug ON creator_embeddings(creator_slug);
CREATE INDEX IF NOT EXISTS idx_ce_embedding_hnsw ON creator_embeddings
    USING hnsw ((embedding::halfvec(3072)) halfvec_cosine_ops);

-- Error tracking on bot_configs
ALTER TABLE bot_configs ADD COLUMN IF NOT EXISTS error_message TEXT;
ALTER TABLE bot_configs ADD COLUMN IF NOT EXISTS provisioning_status TEXT DEFAULT NULL;
```

- Deploy operator service, check Railway logs for `init_db OK` on both new tables
- Connect to Railway Postgres and confirm tables exist: `\dt creator_configs` and `\dt creator_embeddings`

## Phase 2 — Multi-Tenant Retriever
**Time: ~2 hours | Risk: Low | Touches: `app/retrieval/pg_retriever.py`, `app/brain/handler.py`**

### Step 2A — Build `app/retrieval/pg_retriever.py`

New file. Implements the existing `BaseRetriever` interface exactly. On each query:
1. Embed the query text via Gemini embedding API (same model as ingestion: `gemini-embedding-001`, 3072 dims)
2. Run `SELECT chunk_text FROM creator_embeddings WHERE creator_slug = %s ORDER BY embedding <=> %s LIMIT k`
3. Return the chunk texts

Scoped by `creator_slug` — a retriever for `'haley'` cannot return Zarna's data.

### Step 2B — Update `create_brain()` in `app/brain/handler.py`

Accept an optional `slug` parameter. When slug is provided, pass `PgRetriever(slug)` to the brain. When no slug is provided, fall back to the existing `EmbeddingRetriever` — **Zarna stays unchanged at this point.**

## Phase 3 — Provisioning Module
**Time: ~4-6 hours | Risk: Medium | Touches: new `operator/app/provisioning/` module**

This is the core of the whole system. Build and test each sub-module independently before wiring them together.

### Step 3A — `operator/app/provisioning/ingestion.py`

The RAG data pipeline. Builds the knowledge base for any creator.

**Content sources (in priority order):**

| Source | How we get it | Weight in retrieval |
|---|---|---|
| `facts` | `bio` + `extra_context` directly from form | 1.35 (highest) |
| `website_about` | Scrape `/about` page | 1.20 |
| `website_faq` | Scrape `/faq` page | 1.15 |
| `doc_upload` | PDFs / text files uploaded (stored as base64 in request, not on disk) | 1.10 |
| `website_general` | Rest of scraped site pages | 1.00 |
| `podcast` | RSS feed or YouTube channel descriptions | 1.00 |

**How ingestion works:**

```
1. Seed "facts" chunk (always first, highest priority)
   → bio + extra_context written directly as a chunk
   → no scraping needed

2. Scrape website_url
   → requests + BeautifulSoup → extract visible text from homepage, /about, /faq
   → strip nav, footer, boilerplate
   → fallback: if site is JS-heavy and returns empty, log warning and skip

3. Process uploaded files (passed as decoded bytes, never written to disk)
   → PDF → PyPDF2 text extraction from bytes buffer
   → .txt → decode directly

4. (Optional) Fetch podcast/YouTube feed
   → RSS → download episode descriptions + show notes
   → YouTube channel → pull video titles/descriptions via YouTube Data API

5. Chunk all text
   → ~500 tokens per chunk, 50-token overlap
   → keeps semantic units intact across chunk boundaries

6. Batch embed via Gemini embedding API
   → model: gemini-embedding-001, 3072 dimensions (native output)
   → batches of 50 chunks, retry with backoff on rate limit

7. Store in Postgres
   → INSERT INTO creator_embeddings (creator_slug, chunk_text, source, embedding)
   → one row per chunk
   → creator_slug ensures no cross-creator leakage
```

> **Important:** Uploaded files are processed from memory (bytes passed through the request),
> never written to the Railway filesystem. This ensures they survive even if provisioning
> is retried on a new container.

### Step 3B — `creator_config/TEMPLATE_LLM.json`

A clean, machine-fillable blank structure that Gemini populates during `config_writer.py`.

This is **different** from the existing `creator_config/TEMPLATE.json` (which is a human guide for manually building a config). `TEMPLATE_LLM.json` has no human instructions — just the JSON skeleton with placeholder values that Gemini replaces.

The existing `creator_config/TEMPLATE.json` stays as-is for manually-onboarded clients.

### Step 3C — `operator/app/provisioning/config_writer.py`

Generates the AI personality for any creator automatically.

1. Load `creator_config/TEMPLATE_LLM.json` as the output schema
2. Build a Gemini prompt with the creator's `display_name`, `bio`, `tone`, `extra_context`
3. Gemini fills in: `style_rules`, `voice_lock_rules`, `hard_fact_guardrails`, `tone_examples` (5-8 fan/bot pairs)
4. Merge LLM output with safe template defaults
5. Write to `creator_configs` table (Postgres) — **not to disk**

Idempotency: check `SELECT id FROM creator_configs WHERE creator_slug = %s` — if exists, skip.

### Step 3D — `operator/app/provisioning/phone.py`

**Stubbed for now.** Real Twilio logic wired in once A2P campaign is approved.

Stub behavior: returns a fake number like `+15550000001` and logs `"[STUB] phone provisioning skipped"`.

Real behavior (once Twilio approved):
1. Search Twilio API for available US local numbers
2. Purchase one
3. Set its `sms_url` webhook to `/smb/inbound?tenant=<slug>`
4. Add it to the platform messaging service (A2P campaign SID from env)
5. Save the phone number to `operator_users.phone_number`

### Step 3E — `operator/app/provisioning/notifications.py`

One Resend API call. Email includes:
- Their dedicated phone number (or "Coming soon — your number is being assigned" if stub)
- Link to their dashboard
- "First 3 things to do" quick-start guide

### Step 3F — `operator/app/provisioning/__init__.py`

The orchestrator. Runs all steps in sequence, updates status at each step.

```
provision_new_creator(user_id, slug, config):
  set bot_configs.status = 'provisioning'

  → phone.buy_and_configure(slug)        # stubbed until Twilio approved
  → config_writer.generate_and_write(slug, config)
  → ingestion.run(slug, config)
  → notifications.send_welcome(user_id, phone_number)

  set bot_configs.status = 'live'

  On any exception:
    set bot_configs.status = 'provisioning_failed'
    save error_message + traceback to bot_configs.error_message
```

Each step is **idempotent** — if provisioning fails halfway and is retried, completed steps are skipped:
- Phone: check if `operator_users.phone_number` already set → skip
- Config: check if row exists in `creator_configs` for slug → skip
- Embeddings: check if rows exist in `creator_embeddings` for slug → skip
- Email: always resend (safe to repeat)

## Phase 5 — Zarna Migration  ✅ COMPLETE
**Time: ~1 hour | Risk: Low (with verification) | Touches: migration script, `app/retrieval/pg_retriever.py`, `app/retrieval/source_weights.py`, `app/brain/handler.py`, `app/brain/creator_config.py`**

### What was built

1. **`app/retrieval/source_weights.py`** — Extracted Zarna's hand-tuned per-source weights (facts 1.35×, book 1.22×, podcast transcripts 0.74× or excluded, viral skits 1.08-1.18×, etc.) into a reusable `zarna_weight_fn()` factory. `EmbeddingRetriever`'s ranking logic is now shareable with `PgRetriever`.

2. **`app/retrieval/pg_retriever.py`** — Added an optional `weight_fn` constructor param. When set, the retriever does a full per-slug scan in SQL (fetching only `chunk_text + source + distance`, _not_ the 12 KB embedding column), converts distance to similarity, multiplies by `weight_fn(source)`, drops zero-weighted rows (filter semantics), sorts, returns top-K.
   - This is load-bearing for quality: `zarna_facts` chunks need the 1.35× boost to surface, but they don't always rank in the raw top-30 by pure cosine — the full scan guarantees every candidate gets weighted.
   - For Zarna's 3,261 rows, the scan costs ~650 KB of wire + one sort, well under 100 ms.
   - New creators (no weights tuned yet) still take the fast HNSW index path.

3. **`scripts/migrate_zarna_to_pg.py`** — One-off idempotent migration:
   - `creator_config/zarna.json` → `creator_configs` row with `ON CONFLICT DO UPDATE`.
   - `training_data/zarna_embeddings.json.gz` → 3,261 rows into `creator_embeddings` with `slug='zarna'`, preserving the `source` label for weighting.
   - `--apply` required to actually write; dry-run otherwise.
   - Applied to Railway Postgres: 3,261 rows inserted in 28.4 s, config row upserted.

4. **`app/brain/handler.py` / `create_brain(slug=None)`** — Retrieval selection now has three branches:
   - New creator (slug != 'zarna'): `PgRetriever(slug)`, no weights.
   - Zarna + `PG_RETRIEVER_FOR_ZARNA=1` env: `PgRetriever('zarna', weight_fn=zarna_weight_fn(...))`.
   - Zarna otherwise: legacy `EmbeddingRetriever` (Zarna's production default — unchanged).

5. **`app/brain/creator_config.py` / `load_creator`** — Added Postgres fallback so dynamically provisioned creators (whose config only exists in the `creator_configs` table, never on disk) get their personality loaded correctly at reply time. Zarna's file still wins when present, so her production deploy is byte-identical.

### Quality verification

`scripts/test_phase5_quality_comparison.py` runs a fixed bank of 12 realistic fan queries through both retrievers and compares top-7 chunks. Thresholds: ≥60 % top-1 exact match, ≥50 % average Jaccard.

**Actual result: 100 % top-1, 100 % Jaccard across all 12 queries.** `PgRetriever('zarna')` with `zarna_weight_fn()` is byte-for-byte equivalent to `EmbeddingRetriever`. Safe to flip `PG_RETRIEVER_FOR_ZARNA=1` in production whenever you want to cut the file path.

## Phase 6 — Local Chat Simulator  ✅ COMPLETE
**Use it throughout all phases.**

### What was built

`scripts/chat_local.py` — an in-process REPL that builds a brain for any slug via `create_brain(slug)` and calls `brain.handle_incoming_message` directly. Same code path as `/message`, but with per-slug selection the production endpoint doesn't expose (production `/message` is hardwired to Zarna's global brain).

Why in-process instead of HTTP:
  - The production `/message` endpoint creates one global brain at import time and doesn't accept a slug parameter. It can only talk to Zarna. Going through it for multi-tenant local testing would require either spinning up a separate Flask server per slug, or changing a load-bearing production endpoint.
  - Calling `brain.handle_incoming_message(phone, text)` directly is byte-for-byte what `/message` does internally after auth — we lose nothing in fidelity and gain full slug flexibility.

### Usage

```bash
python scripts/chat_local.py                    # Zarna via legacy EmbeddingRetriever (= production default)
python scripts/chat_local.py --slug zarna       # Zarna via the new slug-aware path
PG_RETRIEVER_FOR_ZARNA=1 python scripts/chat_local.py --slug zarna   # Zarna via PgRetriever + zarna_weight_fn
python scripts/chat_local.py --slug haley       # any newly provisioned creator
python scripts/chat_local.py --slug haley --phone +15551234567       # pin the fake fan number
```

### Slash-commands inside the REPL

| Command | What it does |
|---|---|
| `/exit` | quit |
| `/help` | list commands |
| `/reset` | wipe this fake fan's conversation history + memory across all tables (messages, contacts, fan_memory, conversation_sessions, fan_reply_context). Useful for replaying a flow from a clean state. |
| `/new` | roll a new random `+1555…` fake phone so subsequent messages land on a fresh fan |
| `/history [n]` | dump the last N messages (default 20) as stored in Postgres |
| `/slug <new_slug>` | hot-swap the brain to a different creator without restarting the REPL. `/slug none` returns to Zarna's legacy path. |
| `/phone` | show the phone number in use |
| `/debug on\|off` | toggle DEBUG logging for the next message (retrieval, routing, generation traces) |

### Verified behaviour
- [x] Boots a Zarna legacy brain (`EmbeddingRetriever`) and generates in-voice replies
- [x] Boots a Zarna slug-aware brain (`PgRetriever` + `zarna_weight_fn`) when `PG_RETRIEVER_FOR_ZARNA=1` — same quality
- [x] Hot-swaps slug mid-session via `/slug` without restart
- [x] `/reset` deletes rows from `messages`, `contacts`, `fan_memory`, `conversation_sessions`, `fan_reply_context` for the current phone (verified: `/history` after `/reset` shows "no messages yet")
- [x] `/history` renders the on-DB transcript with `[you]` / `[bot]` markers
- [x] Per-message latency is printed (real round-trip, 5-15s per reply via live Gemini)

## How the Bot Works at Runtime (Post-Build)

```
Fan texts +15551234567 "when's your next show?" →

Twilio webhook → /smb/inbound?tenant=haley →
  Load CreatorConfig from creator_configs WHERE creator_slug='haley'
  PgRetriever('haley').get_relevant_chunks("when's your next show?")
    → embed query via Gemini
    → SELECT chunk_text FROM creator_embeddings
        WHERE creator_slug = 'haley'
        ORDER BY embedding <=> query_vector LIMIT 7
    → returns Haley's most relevant content chunks
  generate_reply(intent, chunks, creator_config=haley_config)
    → system prompt uses Haley's display_name, style_rules, tone_examples
    → context is Haley's actual data from her site/docs
    → reply sounds like Haley, answers with Haley's information

No Zarna data is touched. No Zarna config is loaded. The slug scopes everything.
```

## Risks and Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Zarna quality degrades after pgvector migration | Medium | Migrate existing embeddings as-is (no re-embedding). Run 20+ query comparison tests before flipping switch. Keep `.json.gz` as backup for 30 days. |
| Gemini generates a weak or generic personality | Medium | Auto-generate first, then human reviews config before marking status `live`. Add a "review" step to the dashboard. |
| Scraper fails on JS-heavy sites | Medium | Log warning, skip gracefully, fall back to bio/extra_context chunks only. Document that manual content upload always works. |
| Uploaded files lost on Railway restart | Solved | Files processed from memory (bytes), never written to filesystem. Postgres stores everything. |
| Creator data cross-contamination | Low | Every DB query uses `WHERE creator_slug = %s`. Integration test verifies HaleyBot can't retrieve Zarna's chunks. |
| Embedding costs grow unexpectedly | Low | Batch carefully, dedupe before embedding, track cost per provisioning job. ~$0.002 per creator at 400 chunks. |
