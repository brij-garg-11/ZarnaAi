# Bot Creation Pipeline
_How a new creator goes from sign-up to a live, personalized AI texting bot_
_Last updated: Apr 21, 2026_

---

## Overview

The goal is a fully self-serve pipeline where any performer or business fills out a form, and within ~70 seconds they have a dedicated phone number with an AI bot that sounds like them, knows their content, and can answer fan questions using their actual data.

Zarna is the proof of concept. Every new creator (HaleyBot, etc.) follows this same pipeline automatically.

---

## What Gets Created Per Creator

```
operator_users row
  └── creator_slug, account_type, phone_number (filled after provisioning)

bot_configs row
  └── config_json (from onboarding form), status, error_message

creator_config/<slug>.json
  └── AI personality file — voice, style rules, guardrails, links
      Generated by LLM from their form data

creator_embeddings (Postgres table, pgvector)
  └── All their content as vector chunks
      Scraped from website, uploaded docs, podcast feeds, bio text

contacts table
  └── Empty at start — fans added as they text in
```

---

## Status Lifecycle

```
submitted → provisioning → live
                ↓
         provisioning_failed  (error_message stored, retry available)
```

| Status | What it means | Frontend shows |
|---|---|---|
| `submitted` | Form saved, job not started yet | "We're reviewing your submission" |
| `provisioning` | Background job running | "Setting up your bot…" spinner |
| `live` | Everything done, bot is active | Dashboard with phone number |
| `provisioning_failed` | A step broke | "Setup hit a snag — retry" button |

## The Full Pipeline, Step by Step

### Step 1 — Creator fills out onboarding form

The Lovable frontend sends `POST /api/onboarding/submit` with:

| Field | What it is |
|---|---|
| `display_name` | Their name ("Haley Johnson") |
| `bio` | Short description of who they are |
| `tone` | How they want the bot to sound (funny, warm, hype, etc.) |
| `website_url` | Their main site — we'll scrape this |
| `podcast_url` | Optional — RSS feed or YouTube channel |
| `media_urls` | Any other links (YouTube, Spotify, etc.) |
| `extra_context` | Free text — anything they want the bot to know |
| `uploaded_files` | Optional PDFs, transcripts, bios |

**What happens in the route (`operator/app/routes/api.py`):**
1. Validates input
2. Generates `creator_slug` from their name (e.g. `haley`)
3. Inserts row into `bot_configs` with `status='submitted'`
4. Fires background thread → `provision_new_creator(user_id, slug, config_json)`
5. Returns `200 OK` immediately — provisioning runs async

---

### Step 2 — Background provisioning job kicks off

`operator/app/provisioning/__init__.py` — the orchestrator.

```
provision_new_creator(user_id, slug, config):
  set status = 'provisioning'
  → phone.buy_and_configure(slug)
  → config_writer.generate_and_write(slug, config)
  → ingestion.run(slug, config)
  → notifications.send_welcome(user_id, phone_number)
  set status = 'live'

  On any failure:
    set status = 'provisioning_failed'
    save error_message to bot_configs
```

Each step is **idempotent** — if provisioning fails halfway and is retried, already-completed steps are skipped:
- Phone: check if `operator_users.phone_number` already set → skip
- Config: check if `creator_config/<slug>.json` already exists → skip
- Embeddings: check if `creator_embeddings` rows exist for slug → skip
- Email: always resend (safe to repeat)

---

### Step 3 — Buy and wire a Twilio phone number

`operator/app/provisioning/phone.py`

1. Search Twilio API for available US local numbers
2. Purchase one
3. Set its `sms_url` webhook to `/smb/inbound?tenant=<slug>`
4. Add it to the platform messaging service (A2P campaign)
5. Save the phone number to `operator_users.phone_number`

**Returns:** phone number string (e.g. `+15551234567`)

> **Dependency:** Requires `TWILIO_MESSAGING_SERVICE_SID` in env — the platform A2P campaign SID. Currently pending Twilio approval (Low Volume Mixed campaign). Once approved, this unblocks auto-provisioning entirely.

---

### Step 4 — Generate the personality config file

`operator/app/provisioning/config_writer.py`

This is what makes HaleyBot sound like Haley instead of Zarna.

1. Load `creator_config/TEMPLATE.json` (blank personality template)
2. Build a prompt to Gemini with the creator's `display_name`, `bio`, `tone`, `extra_context`
3. Gemini fills in:
   - `style_rules` — how to write, what register to use, what to avoid
   - `voice_lock_rules` — facts the bot must never get wrong (family names, relationships, opinions)
   - `hard_fact_guardrails` — things the bot should never say or claim
   - `tone_examples` — 5-8 sample fan/bot exchanges in their voice
4. Merge LLM output with template defaults
5. Write to `creator_config/<slug>.json`

The brain (`ZarnaBrain` / equivalent) loads this file at startup via `load_creator(CREATOR_SLUG)`. The entire personality — style, voice, links, guardrails, example replies — comes from this file. Nothing is hardcoded for new creators.

---

### Step 5 — Scrape, chunk, embed, and store their content

`operator/app/provisioning/ingestion.py`

This is the RAG data pipeline. It builds the knowledge base the bot draws from when answering questions.

**Content sources (in priority order):**

| Source | How we get it | Weight in retrieval |
|---|---|---|
| `facts` | `bio` + `extra_context` directly from form (no scraping) | 1.35 (highest) |
| `website_about` | Scrape their `/about` page | 1.20 |
| `website_faq` | Scrape their `/faq` page | 1.15 |
| `doc_upload` | PDFs / text files they uploaded | 1.10 |
| `website_general` | Rest of scraped site pages | 1.00 |
| `podcast` | RSS feed or YouTube channel (if provided) | 1.00 |

**How ingestion works:**

```
1. Scrape website_url
   → requests + BeautifulSoup → extract visible text from homepage, /about, /faq
   → strip nav, footer, boilerplate

2. Process uploaded files
   → PDF → PyPDF2 text extraction
   → .txt → read directly

3. Seed "facts" chunk
   → bio + extra_context written directly as a high-priority chunk
   → no scraping needed, always done first

4. (Optional) Fetch podcast feed
   → RSS → download episode descriptions + show notes
   → YouTube channel → pull video titles/descriptions

5. Chunk all text
   → ~500 tokens per chunk, 50-token overlap
   → keeps semantic units intact across chunk boundaries

6. Batch embed via Gemini embedding API
   → same model used at query time (gemini-embedding-001, 768 dimensions)
   → batches of 50 chunks, retry on rate limit

7. Store in Postgres
   → INSERT INTO creator_embeddings (creator_slug, chunk_text, source, embedding)
   → one row per chunk
   → creator_slug column ensures no cross-creator leakage
```

---

### Step 6 — Send welcome email

`operator/app/provisioning/notifications.py`

One Resend API call. Email includes:
- Their dedicated phone number
- Link to their dashboard
- "First 3 things to do" quick start guide

---

### Step 7 — Fan texts in, bot replies

When a fan texts the creator's number:

```
Fan texts +15551234567 "when's your next show?" →

Twilio webhook → /smb/inbound?tenant=haley →
  SMB brain loads CreatorConfig from creator_config/haley.json
  PgRetriever('haley').get_relevant_chunks("when's your next show?")
    → embeds the query via Gemini
    → SELECT chunk_text FROM creator_embeddings
      WHERE creator_slug = 'haley'
      ORDER BY embedding <=> query_vector LIMIT 7
    → returns Haley's most relevant content chunks
  generate_reply(intent, chunks, creator_config=haley_config)
    → system prompt uses Haley's name, style rules, voice examples
    → context is Haley's actual data from her site/docs
    → reply sounds like Haley, answers with Haley's information
```

No Zarna data is touched. No Zarna config is loaded. The slug scopes everything.

## How the RAG Retrieval Works at Runtime

The `BaseRetriever` interface (`app/retrieval/base.py`) allows any implementation — file-based or DB-backed. Two implementations exist (one built, one to build):

**`EmbeddingRetriever`** (current, Zarna-only):
- Loads a `.json.gz` file of pre-built embeddings at startup
- Cosine similarity in Python against all chunks in memory
- Works great for one creator; doesn't scale to many

**`PgRetriever`** (to build, multi-tenant):
- Implements the same `BaseRetriever` interface
- On each query: embed query → pgvector `<=>` operator → `LIMIT k`
- Scoped by `creator_slug` — impossible to retrieve another creator's data
- `create_brain()` passes a `PgRetriever(slug)` instead of `EmbeddingRetriever`

The brain calls `self.retriever.get_relevant_chunks(message)` — it doesn't know or care which retriever it is. Zero changes to brain logic when swapping.

## New DB Table Required

```sql
-- Enable pgvector (one-time, run on Railway Postgres console)
CREATE EXTENSION IF NOT EXISTS vector;

-- Multi-tenant RAG embeddings
CREATE TABLE IF NOT EXISTS creator_embeddings (
    id           BIGSERIAL PRIMARY KEY,
    creator_slug TEXT NOT NULL,
    chunk_text   TEXT NOT NULL,
    source       TEXT NOT NULL DEFAULT 'general',
    embedding    vector(768),
    created_at   TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_ce_slug ON creator_embeddings(creator_slug);
CREATE INDEX IF NOT EXISTS idx_ce_embedding ON creator_embeddings
    USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);

-- Add error tracking to bot_configs
ALTER TABLE bot_configs ADD COLUMN IF NOT EXISTS error_message TEXT;
```

## Zarna Migration Plan (existing creator → new system)

Once pgvector is live and `PgRetriever` is built:

1. Run one-time migration script: read `training_data/zarna_chunks.json` → `INSERT INTO creator_embeddings WHERE creator_slug='zarna'`
2. Change `create_brain()` to use `PgRetriever('zarna')` instead of `EmbeddingRetriever`
3. Run test queries, compare top-k chunks to old retriever — verify quality is identical
4. Keep `zarna_embeddings.json.gz` as backup for 30 days, then delete

Zarna becomes just another slug. No special casing.

## End-to-End Timeline (once built)

```
0 sec   Creator submits onboarding form
        → DB: bot_configs.status = 'submitted'
        → Background thread starts

1 sec   Thread running
        → DB: status = 'provisioning'
        → phone.py: buys +1(XXX)XXX-XXXX, wires webhook (~2 sec)

5 sec   → config_writer.py: Gemini generates personality JSON (~5 sec)
        → creator_config/haley.json written to disk

10 sec  → ingestion.py: scrape site, fetch feeds, process uploads
          chunk text, batch embed via Gemini (~55 sec for typical site)
          INSERT ~200-500 rows into creator_embeddings

65 sec  → notifications.py: "You're live" email sent (~1 sec)
        → DB: status = 'live'

~70 sec Creator sees dashboard with their phone number
        Bot is live and ready to receive texts
```
