1. Configurable Voice & Persona
The biggest blocker. The AI sounds like Zarna because her voice is hardcoded into Python.
MUST BUILD Nothing works for another creator without this. Every reply will still sound like Zarna.

The Problem — Where It Lives in Code

In app/brain/generator.py, the AI voice is a hardcoded Python string called _STYLE_RULES. It references Zarna by name, her topics (Indian-mom, mother-in-law, Shalabh, Baba Ramdev), her specific openers, and her banned words. Below that is _TONE_EXAMPLES — real transcripts of Zarna's replies used to tune the tone. These are not configurable. A new client's AI will inherit all of this.

# app/brain/generator.py ← what exists today (lines 57–88) _STYLE_RULES = """ Voice: sharp, high-energy, opinionated, family- and culture-aware — conversational stand-up energy, never generic or male-coded. Prefer parenting, marriage, immigrant-family, Indian-mom angles... Exception: if topic is Shalabh / husband / mother-in-law / Baba Ramdev... """
# app/config.py ← training data paths are also hardcoded to zarna CHUNKS_PATH = os.path.join(_BASE_DIR, "training_data", "zarna_chunks.json") EMBEDDINGS_PATH = os.path.join(_BASE_DIR, "training_data", "zarna_embeddings.json.gz")

The Fix — What to Build

Two things: (1) make _STYLE_RULES and _TONE_EXAMPLES loadable from a file on disk, (2) make the training data paths use a configurable CREATOR_SLUG. New creator = new config file + new training data files. Zero code changes per client.

1

Create a creator config file format

A JSON file per creator that holds their voice rules, tone examples, display name, and what topics to avoid. Stored as creator_config/[creator_slug].json

2

Add one env var: CREATOR_SLUG

Each client deployment sets CREATOR_SLUG=sarah. The app loads creator_config/sarah.json at startup. Training data becomes training_data/sarah_chunks.json and sarah_embeddings.json.gz.

3

Modify generator.py to load from config

Replace the hardcoded strings with a loader that reads from the JSON file. Falls back to default values if nothing is set, so Zarna's deployment is unchanged.

# What the fix looks like in app/config.py CREATOR_SLUG = os.getenv("CREATOR_SLUG", "zarna") CHUNKS_PATH = os.path.join(_BASE_DIR, "training_data", f"{CREATOR_SLUG}_chunks.json") EMBEDDINGS_PATH = os.path.join(_BASE_DIR, "training_data", f"{CREATOR_SLUG}_embeddings.json.gz") CREATOR_CONFIG_PATH = os.path.join(_BASE_DIR, "creator_config", f"{CREATOR_SLUG}.json")
# What the fix looks like in app/brain/generator.py import json, os from app.config import CREATOR_CONFIG_PATH def _load_creator_config(): try: with open(CREATOR_CONFIG_PATH) as f: return json.load(f) except (FileNotFoundError, json.JSONDecodeError): return {} _creator = _load_creator_config() _STYLE_RULES = _creator.get("style_rules", _DEFAULT_STYLE_RULES) _TONE_EXAMPLES = _creator.get("tone_examples", _DEFAULT_TONE_EXAMPLES)

What the Creator Config File Looks Like

You fill this out for each new client during onboarding. It becomes the soul of the AI for that creator.

// creator_config/sarah.json { "name": "Sarah", "style_rules": "Voice: dry, self-deprecating, midwestern chaos energy...", "tone_examples": "Fan: loved the show!\nSarah: Honestly thank you — my therapist will be thrilled to hear it.", "banned_words": ["honey", "darling"], "off_limits_topics": ["my ex", "politics"], "signature_phrases": ["not gonna lie", "anyway"] }

The style_rules field is the most important — it's the full voice guide. Writing a good one for a new client takes 1–2 hours and requires reading/watching their material.

2. Training Data Onboarding Pipeline
The scripts already exist. What's missing is a single entry point that runs them all for a new creator.
MUST BUILD Without training data, the AI gives generic answers. The pipeline exists but is manual and fragmented.

What Already Exists

All the ingestion scripts are in scripts/. They work — you used them all for Zarna.

ScriptWhat it ingestsInput needed
ingest_youtube.pyYouTube video transcriptsChannel URL or video list
ingest_podcast.pyPodcast episodes (audio → transcript)RSS feed URL or episode files
ingest_special.pyStand-up special transcriptText or PDF file
ingest_book.pyBook chaptersPDF or text file
ingest_skits.pyShort video skitsFile or URL list
build_embeddings.pyTurns all chunks into vectorsRuns after all ingestion

What's Missing

There's no unified entry point. To onboard a new creator today, you'd have to remember which scripts exist, run each manually, figure out the right arguments, and hope nothing breaks midway. There's also no way to see "what did I already ingest for this creator?"

What to Build: scripts/onboard_creator.py

One script that reads a creator's intake form (a JSON file you fill out after signing a client), runs all ingestion steps in the right order, and builds the embeddings at the end. Run it once and the creator is ready to deploy.

1

Create intake form template: creator_config/TEMPLATE.json

A commented JSON template that documents every field. You copy it for each new client, fill it out, and the script reads it.

2

Write onboard_creator.py — reads the config, runs everything

Calls each ingestion script for the content sources listed in the config. Reports how many chunks were created from each source. Stops and tells you clearly if something fails.

3

Output a "knowledge health" report

After running, prints a summary: X chunks from YouTube, Y from podcast, Z from special. Flags if any category is empty (e.g. no YouTube content). Tells you the AI will be weaker in that area.

# What running it looks like $ python scripts/onboard_creator.py --creator sarah # Output: Loading creator config: creator_config/sarah.json → YouTube (3 videos) 412 chunks ✓ → Podcast (12 episodes) 1,840 chunks ✓ → Special (1 file) 280 chunks ✓ → Book 0 chunks ⚠ no book path configured Building embeddings... 2,532 total chunks Done. Saved: training_data/sarah_chunks.json training_data/sarah_embeddings.json.gz

The Intake Config Template

3. Deployment Template
Most of the config is already env-var driven. What's missing is a documented, repeatable process.
MUST BUILD Right now "deploying for a new client" lives in your head. It needs to be a checklist you can hand off.

What Already Works

app/config.py is almost entirely env-var driven. A new client deployment would just need a Railway project with different env vars set. The hard parts are already abstracted — you just need to document them.

The only hardcoded part is the voice/training data (which Build #1 fixes).

What to Create: .env.client.example

A documented env var template for a new client deployment. Every variable with a comment explaining what it is and where to get the value.

# ── REQUIRED for every deployment ────────────────────────────── CREATOR_SLUG=sarah # matches creator_config/sarah.json DATABASE_URL=postgresql://... # Railway Postgres plugin → Database URL SECRET_KEY=... # 32+ random chars, generate with: openssl rand -hex 32 # ── AI ────────────────────────────────────────────────────────── GEMINI_API_KEY=... # console.cloud.google.com OPENAI_API_KEY=... # optional but improves conversational replies ANTHROPIC_API_KEY=... # optional but improves nuanced/advice replies # ── SMS: SlickText (new accounts) ──────────────────────────────── SLICKTEXT_API_KEY=... # Settings → API & Webhooks SLICKTEXT_BRAND_ID=... # same page as API key SLICKTEXT_WEBHOOK_SECRET=... # a long random string; set in SlickText webhook config too # ── Operator dashboard login ───────────────────────────────────── OPERATOR_BOOTSTRAP_EMAIL=sarah@example.com OPERATOR_BOOTSTRAP_PASSWORD=... # ── Optional ──────────────────────────────────────────────────── TWILIO_ACCOUNT_SID=... TWILIO_AUTH_TOKEN=... TWILIO_PHONE_NUMBER=+1...

What to Create: docs/deploy_checklist.md

A step-by-step checklist for spinning up a new client. Every step is actionable with exact commands or URLs. Nothing lives in your head.

1

Complete the intake form

Fill out creator_config/[slug].json with voice rules and content sources.

2

Run the onboarding pipeline

python scripts/onboard_creator.py --creator [slug] — review the knowledge health report.

3

Set up SMS account

4

Hit /operator/setup once to create the initial login. Then invite the creator as owner, add your team as operators.

7

Creator signs off

8

Auth & Team Management
Already done — and better than you might think.
ALREADY DONE The operator app has email/password login, named accounts, and owner/operator roles.

What's Built

  • Email + password login with hashed passwords (Werkzeug pbkdf2)
  • Owner vs Operator roles — owner can add/remove team members
  • Bootstrap flow: hit /operator/setup to create the first account on a fresh deploy
  • Deactivate users without deleting (audit trail preserved)
  • Password reset by owner
  • last_login_at tracking per user

How It Works for a New Client

Each client deployment is isolated — its own Railway project, its own database, its own operator_users table. Setup looks like:

1

Bootstrap email/password → creator becomes owner

2

You (Brij's team) get added as operators

3

Creator can add their own manager/team as operators too

This already works — no changes needed for multi-client.

Relevant Files

operator/app/routes/auth.py — login, logout, setup, session management
operator/app/routes/team.py — add/deactivate/reset users, toggle owner
operator/app/templates/login.html — login page
operator/app/templates/team.html — team management UI

Blast Workflow
The blast tool is fully built — drafts, test send, scheduling, image upload, link tracking.
ALREADY DONE One of the most complete features in the platform. Supports Twilio and SlickText, draft saving, scheduled sends.

What's Built

  • Draft board — compose and save before sending
  • Audience targeting: all subscribers, by tag, by location, by show list, random sample %
  • Send a test to your own phone before blasting
  • Confirmation checkbox required before live send
  • Scheduled sends (background scheduler)
  • Image / MMS support (stored in Postgres, served as public URL)
  • Link tracking with click counts per blast
  • Clone a sent blast for resend
  • Blast status polling (sent / sending / failed counts)
  • Stuck send recovery (resets drafts stuck in "sending" after 5 min)

TCPA Note — What You Still Need

The technical blast safeguards are solid. What's missing is a paper trail of client approval for each blast.

Right now the blast tool has a confirmation checkbox, but there's no record of who confirmed or when. For liability purposes — especially if a blast goes to 10K+ people — you want an email/Slack confirmation step that creates an audit trail outside the app.

Simple fix: require the "owner" role (creator) to send, and log who pressed send with a timestamp. The created_by field in the database already captures this email — it just needs to be more visible.

Relevant Files

operator/app/routes/blast.py — all blast routes (draft, send, schedule, clone, image upload)
operator/app/blast_sender.py — background thread that executes sends
operator/app/templates/blast.html — compose UI

Live Shows & Streams
The live show system is complete — keyword signup, event types, timezone handling, join confirmations.
ALREADY DONE The battle-tested feature — used at the Rhode Island show that drove 323 signups in one night.

What's Built

  • Create shows with keyword, time window, event type (comedy / live stream / other)
  • Draft → Live → Ended lifecycle with one-at-a-time enforcement
  • Auto-welcome texts: rotating comedy set vs. streaming-themed set vs. silent join
  • Typo forgiveness on longer keywords (fuzzy matching)
  • Keyword-only joins suppress the AI so fans don't get a chat reply on that text
  • Timezone-aware time windows (creator enters in their local time)
  • Show signups list saved permanently — downloadable CSV
  • Recent actions audit log per show

One Thing to Watch

The welcome texts for comedy shows and live streams are hardcoded in app/live_shows/join_confirmations.py. They work great for Zarna. For a new creator, they'll need to be either:

  • Made configurable (add to the creator config JSON), or
  • Manually updated per deployment before launch

This is a lower-urgency fix — the messages are generic enough to work for most performers. But a creator who wants their own welcome text will notice.

Relevant Files

app/live_shows/signup.py — signup logic, join confirmation trigger
app/live_shows/join_confirmations.py — the welcome text templates (hardcoded, consider making configurable)
app/live_shows/keyword_match.py — fuzzy keyword matching
operator/app/routes/shows.py — dashboard UI routes

4. Creator Access Role
Creators need to get into the dashboard without being able to accidentally break their own setup.
SHOULD BUILD The current Owner/Operator model works but gives creators more access than they need.

The Problem

Right now there are two roles: Owner (can manage team, change passwords, full access) and Operator (everything except team management). When you give a creator the "Owner" role, they can see your team members, manage them, and reset passwords. When you give them "Operator," they can't create their own team.

This is a medium-urgency problem. For most creators, just making them "Operator" and adding your team as Owners is fine in the short term.

What to Build

Add a Creator role to operator/app/routes/auth.py and operator/app/routes/team.py. This role can:

FeatureCreatorOperator (you)Owner
View dashboard / stats✓✓✓
View conversations✓✓✓
Create & manage shows✓✓✓
Send blasts✓ (with confirm)✓✓
Export audience CSV✓✓✓
Add team members✗✗✓
Reset passwords✗✗✓

Owner stays as you (Brij). Creator gets full operational access but can't touch the team management. Minimal code change — just add is_creator boolean to the operator_users table and check it on the team routes.

5. Monthly Reporting
Clients who don't log in every day need a reason to remember the platform is working.
SHOULD BUILD This is a retention play. Clients who don't see numbers forget the platform is delivering value.

What to Build

A scheduled monthly email (or PDF) auto-generated from the database with the numbers that matter to a creator. Uses the scheduler that already exists in operator/app/scheduler.py.

What the Report Should Include

  • Total subscribers (and change from last month)
  • Engagement rate — active fans / total
  • Messages sent by fans this month
  • New subscribers from shows — by show name
  • Blast performance — how many received, if any were sent
  • Top 3 things fans asked about (intent breakdown)
  • One line: "Your fans are talking to you X times per week on average"

Implementation

Two options, in order of effort:

A

Simple: email with stats table

Add a monthly cron job to the scheduler. Runs a query, formats stats as plain text email, sends via SendGrid or Postmark. Low effort, totally sufficient.

B

Better: add a /operator/report page

A dedicated "monthly wrap" route that renders the stats as a clean HTML page. Can be linked from email or bookmarked. Same data, better presentation.

6. AI Quality Flags
At scale, the AI will occasionally say something off-brand. You need a way to catch it.
SHOULD BUILD Without this, a bad reply can live in a fan's inbox for weeks before anyone notices.

What to Build

Add a flag button to the conversation thread view in the operator dashboard. When a team member clicks it on a specific assistant message, it writes to a flagged_messages table with the message ID, reason, and who flagged it.

This is your QA loop. You review flagged messages weekly. Patterns tell you when to update the voice config, add a banned topic, or adjust a style rule. Without this, you have no feedback signal.

What Gets Flagged

  • Reply that sounded nothing like the creator
  • Reply that was factually wrong (wrong show date, wrong book title)
  • Reply that was off-brand or too generic
  • Reply that hit a topic the creator wanted to avoid
  • Reply that felt robotic or overly formal

Minimal Implementation

1

Add flagged_messages table to DB

Columns: message_id, flagged_by, reason, created_at

2

Add flag button to conversation thread UI

Small icon next to each assistant message. Click → modal with "why is this wrong?" dropdown.

3

Add /operator/quality route

A simple list of all flagged messages with their conversations. Review weekly.

7. Knowledge Base Refresh Workflow
When a creator drops new material, the AI needs to learn it without breaking what's already there.
NICE TO HAVE Not urgent for launch, but becomes necessary once creators start touring or releasing new content.

The Problem

Right now, adding new content to the knowledge base means re-running ingestion scripts and rebuilding embeddings. If you do it wrong, you could overwrite existing chunks. There's no "add to" flow — only "rebuild from scratch." For a creator who releases a new special every year and a new podcast episode every week, this is a real operational burden.

What to Build

1

Add an incremental flag to ingestion scripts

Instead of rebuilding the full chunk file, append new chunks to the existing one. Track which URLs/files have already been ingested in a small manifest file (training_data/[slug]_manifest.json) so nothing is ingested twice.

2

Update embeddings incrementally

Only embed the new chunks, then merge into the existing .json.gz. A full rebuild takes minutes; this takes seconds.

3

(Optional) Add an /operator/knowledge-base page

Shows the creator what their AI knows: chunk count by source, last updated date per source, and a button to trigger a refresh. Mostly useful as trust-building — lets creators see the AI isn't making things up.

8. Multi-Client Hub (Your Operator View)
Once you have 3+ clients, you'll want a single view to monitor all of them.
NICE TO HAVE Not needed until you're managing multiple active deployments simultaneously.

The Problem

Today there's no "Brij's view of all clients." If you have 5 client deployments, you'd have to log into 5 different dashboards to see if anything is broken. One deployment could be down and you wouldn't know until the client complains.

What to Build: A Hub Dashboard

A separate internal app (or a new route on one of your existing apps) that queries all client deployments and shows you a health overview.

ColumnWhat It Shows
Client nameCreator slug / display name
StatusGreen / Yellow / Red based on /health endpoint
SubscribersTotal fan count across all lists
Last activityWhen the most recent inbound text was received
Errors (24h)AI errors, webhook auth failures from ops_metrics
Quick link→ opens that client's operator dashboard

This can be as simple as a static HTML page that fetches each client's /health endpoint and their stats API. No separate database needed.
